# Тестовое задание

**Интеллектуальный менеджер задач с интеграцией LLM**

---

## 1. Описание задачи

### 1.1 Обзор проекта

Вам предлагается создать **Интеллектуальный менеджер задач** — веб-приложение для управления делами (To-Do List) с интегрированным LLM-ассистентом.
Приложение должно позволять пользователям эффективно управлять своими повседневными задачами, одновременно используя возможности искусственного интеллекта
для повышения производительности. Основная идея состоит в том, что LLM выступает в качестве умного помощника, который помогает пользователям работать с задачами эффективнее: например, предлагая приоритеты задач, автоматически категоризируя их,
декомпозируя сложные задачи на подзадачи или отвечая на вопросы о рабочей нагрузке.

### 1.2 Пользовательские истории

Следующие пользовательские истории определяют минимальный функциональный объём приложения. Каждая история описывает функционал с точки зрения конечного пользователя. Кандидаты должны реализовать все перечисленные ниже истории, а по желанию могут добавить дополнительные функции.

| ID | Функция | Описание |
|----|---------|----------|
| US-1 | CRUD задач | Как пользователь, я могу создавать, читать, обновлять и удалять задачи. Каждая задача имеет название, описание, приоритет (низкий/средний/высокий), статус (ожидает/в работе/готово), срок выполнения и время создания. |
| US-2 | Фильтрация и поиск | Как пользователь, я могу фильтровать задачи по статусу, приоритету и сроку. Также я могу искать задачи по названию или описанию через поле текстового поиска. |
| US-3 | LLM: Умная категоризация | Как пользователь, я могу нажать кнопку, чтобы LLM автоматически предложил категорию или тег для задачи на основании её названия и описания. Я могу принять или отклонить предложение. |
| US-4 | LLM: Декомпозиция задачи | Как пользователь, я могу попросить LLM разбить сложную задачу на более мелкие, выполнимые подзадачи. Сгенерированные подзадачи появляются как предложения, которые я могу принять или изменить. |
| US-5 | LLM: Предложение приоритета | Как пользователь, я могу попросить LLM предложить уровень приоритета для задачи на основании её описания и срока. Предложение появляется встроено с возможностью его применения. |
| US-6 | LLM: Сводка нагрузки | Как пользователь, я могу запросить у LLM естественно-языковую сводку моей текущей рабочей нагрузки, включая информацию о просроченных задачах, ближайших сроках и распределении нагрузки. |

## 2. Технические требования

### 2.1 Рекомендуемый технологический стек

Кандидаты свободны в выборе любого технологического стека, с которым они комфортно работают.

### 2.2 Архитектурные ожидания

Приложение должно следовать чистой, поддерживаемой архитектуре. Хотя мы не предписываем конкретный архитектурный паттерн (MVC, Clean Architecture, микросервисы), база кода должна демонстрировать чёткое разделение ответственности. Следующие принципы должны быть видимы в структуре проекта:

- **Разделение ответственности.** Бизнес-логика, доступ к данным, маршруты API и слой представления должны быть чётко разделены.
- **Проектирование API.** Бэкенд должен экспонировать RESTful API (или GraphQL) с чётко определёнными эндпоинтами, правильными HTTP-методами, соответствующими статус-кодами и консистентными форматами запросов/ответов. Версионирование API — преимущество.
- **Слой данных.** Операции с базой данных должны быть абстрагированы за репозиторием или слоем доступа к данным. Поддержка миграций желательна.
- **Обработка ошибок.** И фронтенд, и бэкенд должны иметь консистентную стратегию обработки ошибок. Ошибки API должны возвращать структурированные ответы. Фронтенд должен корректно отображать ошибки без сбоев или показа стектрейсов.

## 3. Требования к интеграции LLM

Интеграция LLM является дополнением для данного задания и не является обязательной.

### 3.1 Prompt Engineering

Рекоммендации по составлению prompt при работе с LLM:

- Предоставление чётких системных промптов, определяющих роль и ограничения LLM.
- Структурирование промптов с контекстом из данных задач пользователя.
- Указание желаемого формата вывода (JSON, простой текст, markdown).
- Включение примеров (few-shot) там, где это уместно.
- Продуманная обработка случаев, когда LLM возвращает некорректный вывод.

Промпты должны быть спроектированы так, чтобы производить консистентные, парсируемые ответы, пригодные для программной обработки.

### 3.2 Асинхронная обработка и UX

Вызовы LLM API по своей природе медленные, часто занимая несколько секунд.

Рекоммендации:

- **Минимум:** индикатор загрузки (спиннер, skeleton-экран или прогресс-бар).
- **Продвинуто:** потоковая передача ответов (streaming), кеширование частых запросов, дедупликация, очередь фоновой обработки для неинтерактивных функций.

## 4. Функциональные требования

### 4.1 Управление задачами (обязательно)

Все CRUD-операции должны быть полностью функциональными и подключёнными к БД. Задачи поддерживают следующие атрибуты:

| Атрибут | Обязательность | Описание |
|---------|---------------|----------|
| Уникальный идентификатор | Да | Автогенерируемый (UUID или автоинкремент) |
| Название | Да | Непустая строка |
| Описание | Нет | Произвольный текст |
| Приоритет | Да | Низкий / Средний / Высокий |
| Статус | Да | Ожидает / В работе / Готово |
| Срок выполнения | Нет | Дата (date или datetime) |
| Время создания | Да | Автоматическая метка (timestamp) |

Интерфейс должен предоставлять форму для создания/редактирования задач, список или доску для просмотра, диалоги подтверждения для удаления.

### 4.2 Фильтрация и поиск (обязательно)

- Фильтрация по статусу и приоритету (выпадающие меню, переключатели или вкладки).
- Полнотекстовый поиск по названиям и описаниям задач.
- Фильтры должны быть **комбинируемыми** (статус + поиск одновременно).
- Логика фильтрации — **на стороне сервера** (в запросах к БД).

### 4.3 LLM-функции (необязательно)

| Функция | Пользовательская история | Что делает |
|---------|--------------------------|-----------|
| Умная категоризация | US-3                     | LLM предлагает категорию/тег на основе названия и описания |
| Декомпозиция задачи | US-4                     | LLM разбивает сложную задачу на подзадачи |
| Предложение приорета | US-5                     | LLM предлагает приоритет на основе описания и срока |
| Сводка нагрузки | US-6                     | LLM генерирует естественно-языковую сводку по всем задачам |

Каждая функция: чёткий триггер (кнопка/меню) + результат с возможностью принять/изменить/отклонить.
